# MODE 6: 演示 Demo 生成

## 概述

基于已有 PRD 文档，生成一套可在浏览器中直接打开的可交互 HTML Demo 原型页面，涵盖 PRD 中定义的所有页面和弹窗交互。

---

## Step 1: 分析 PRD，提取页面和交互清单

遍历 PRD 中的所有功能点章节，提取以下内容：

1. **页面清单**：识别所有需要的页面
   - 新增页面（PRD 中标注"新增"的页面）
   - 改造页面（PRD 中标注"修改"或"变更"的现有页面）
   - 详情页、列表页、表单页等
   - 页面间的导航关系和跳转逻辑

2. **弹窗/交互清单**：识别所有交互组件
   - Modal 弹窗（确认弹窗、表单弹窗、结果摘要弹窗等）
   - Drawer 侧拉抽屉
   - Toast/Message 提示
   - Popconfirm 气泡确认
   - 选人/选部门等业务组件

3. **输出格式**：生成两张清单表格，向用户确认范围

```markdown
### 页面清单
| # | 页面名称 | 类型 | 所属功能点 | 说明 |
|---|---------|------|-----------|------|

### 弹窗/交互清单
| # | 弹窗名称 | 类型 | 所在页面 | 触发方式 |
|---|---------|------|---------|---------|
```

---

## Step 2: 确认设计资源 & 收集参考素材

向用户一次性询问以下三类参考素材（并行收集，三者同等重要）：

| 素材类型 | 作用 | 询问话术 |
|---------|------|---------|
| **设计系统描述文件** | 提取 CSS 变量/设计 token（颜色、字号、间距、圆角等） | 「项目中是否有设计系统描述文件？（如 design-system.md、主题配置文件）」 |
| **组件库描述文件** | 了解可用组件、API、使用模式 | 「是否有组件库索引或文档？（如 llms.txt、组件库文档链接）」 |
| **线上页面截图** | 理解真实的页面布局、组件排列、视觉风格 | 根据 Step 1 页面清单，精确列出需要截图的页面：「为了更好地还原视觉风格，请提供以下页面的截图：[具体页面列表]」 |

### 素材处理

- **设计系统文件** → 提取 `:root` CSS 变量：品牌色、文字色层级、反馈色、背景色、边框色、间距体系、圆角体系
- **组件库文件** → 了解组件命名规范、常用组件 API（如 Button 的 type/size、Table 的列配置模式）
- **线上截图** → 分析：全局布局结构（Header/Sidebar/Content 比例）、筛选区组件排布、表格样式、操作列按钮风格、状态标签颜色语义
- 同时查找项目中是否已有同类 demo 文件（`.html`），有的话读取作为风格参考

### 兜底策略

如果用户三类素材都无法提供：
1. 明确告知用户：「将使用通用的 Ant Design 风格生成 demo，后续可根据实际系统截图调整」
2. 用户确认接受后再继续
3. 如果使用 Ant Design 风格，阅读 https://ant.design/llms-full.txt 并理解 Ant Design 组件库，在编写 Ant Design 代码时使用这些知识。

---

## Step 3: 生成 Demo 页面

### 技术规范

- **技术栈**：纯 HTML + CSS + JavaScript，无框架依赖，浏览器直接打开
- **前端工具**：可选检查本地是否有 `npx`、`vite`、`live-server` 等工具辅助开发和预览
- **设计 token**：使用 CSS 变量（`:root`）集中管理，便于全局调整

### 页面结构要求

- **必须生成 `index.html`**：导航索引页，卡片式链接到所有子页面，包含 PRD 名称、版本、各页面简要说明
- **每个页面独立完整**：含公共 header/sidebar/layout shell，可单独打开使用
- **页面间可互相跳转**：通过侧边栏导航和面包屑实现

### 数据与交互要求

- **Mock 数据**：硬编码，覆盖正常态、异常态（空状态、错误提示）、边界态（超长文本、大量数据）
- **弹窗交互**：所有弹窗可真实触发和关闭
- **操作反馈**：按钮点击有实际反馈（Toast 提示、状态变更、列表刷新等）
- **表格交互**：支持全选/单选、行 hover 效果
- **状态联动**：操作后状态实际变更（如终止后状态列更新）

### 输出目录约定

```
docs/{PRD所在目录}/demo/
├── index.html           # 导航首页（必须）
├── page-1.html          # 功能页面
├── page-2.html          # 功能页面
└── ...
```

---

## Step 4: 验证与交付

- 在浏览器中打开 `index.html`，逐一验证所有页面和交互
- 检查视觉风格与设计系统/截图的一致性
- 确认所有弹窗可正常打开和关闭
- 确认操作反馈（Toast、状态变更）正常工作
- 输出文件清单 + 交互功能对照表

---

## Step 5: Demo ↔ PRD 双向同步

### 5.1 Demo 修改时反向同步 PRD（Demo → PRD）

当用户要求修改已生成的 Demo 时（调整文案、交互逻辑、功能规则等），在执行修改前先评估影响：

#### 判断规则

| 修改类型 | 是否涉及 PRD | 处理方式 |
|---------|-------------|---------|
| 调整按钮文案、提示文案 | ✅ 是 | 需同步修改 PRD 中对应的交互说明/功能规则 |
| 修改交互流程（如增减弹窗步骤） | ✅ 是 | 需同步修改 PRD 中对应的功能内流程和交互说明 |
| 修改功能逻辑（如变更状态流转） | ✅ 是 | 需同步修改 PRD 中对应的功能规则和验收标准 |
| 调整异常处理方式 | ✅ 是 | 需同步修改 PRD 中对应的异常处理表格 |
| 纯视觉调整（颜色、间距、布局微调） | ❌ 否 | 直接修改 Demo |
| 补充 Mock 数据 | ❌ 否 | 直接修改 Demo |

#### 执行流程

1. 用户提出 Demo 修改需求
2. 判断该修改是否涉及 PRD 中的定义
3. **如果涉及 PRD**：
   - 明确告知用户：「此修改涉及 PRD 中的以下内容，建议一并更新」
   - 列出 PRD 中受影响的具体章节和内容
   - 用户确认后，同时修改 Demo 和 PRD
4. **如果不涉及 PRD**：直接修改 Demo
5. **修改完成后**：无论是否涉及 PRD，均主动输出同步结论：
   - 「已同步更新 PRD [章节] 中的 [内容]」，或
   - 「本次修改为纯 Demo 调整，无需同步 PRD」

---

### 5.2 PRD 修改后正向同步 Demo（PRD → Demo）

当用户在 Demo 已生成后，对 PRD 进行了修改，在 PRD 修改完成后自动评估对 Demo 的影响。

#### 判断规则

| PRD 修改类型 | 是否影响 Demo | 处理方式 |
|------------|-------------|---------|
| 修改交互说明、按钮文案 | ✅ 是 | Demo 中对应页面/弹窗需更新 |
| 修改功能规则、状态流转 | ✅ 是 | Demo 中对应逻辑需更新 |
| 修改异常处理 | ✅ 是 | Demo 中对应提示/行为需更新 |
| 新增或删除功能点/页面 | ✅ 是 | Demo 需新增或删除对应页面 |
| 修改纯文字说明（背景/目标/范围描述） | ❌ 否 | 无需更新 Demo |
| 修改验收标准（无功能变更） | ❌ 否 | 无需更新 Demo |

#### 执行流程

1. PRD 修改完成后，主动检查 Demo 是否存在（`docs/{PRD所在目录}/demo/` 目录下）
2. **如果 Demo 不存在**：跳过，无需处理
3. **如果 Demo 存在**，判断本次 PRD 修改是否影响 Demo：
   - **有影响**：主动询问用户：「此次 PRD 修改影响了 Demo 中的以下内容，是否同步更新 Demo？」，并列出受影响的具体 Demo 页面/交互点
     - 用户选择**立即更新**：执行 Demo 更新，完成后进入 Step 6 更新对应截图
     - 用户选择**稍后更新**：记录待更新项，提示用户可通过「重新生成 Demo」或「更新 Demo」指令触发
   - **无影响**：直接告知用户「本次 PRD 修改不影响 Demo，无需更新」

#### 「更新 Demo」指令

用户可在任意时间通过以下指令触发 Demo 更新：

| 指令 | 行为 |
|------|------|
| `重新生成 demo` / `regenerate demo` | 对照最新 PRD 重新生成全部 Demo 页面 |
| `更新 demo` / `update demo` | 对照最新 PRD，仅更新有变化的页面 |
| `更新 [页面名] 的 demo` | 仅更新指定页面 |
| `根据 PRD 变更调整 demo` | 推断 PRD 变更范围，更新受影响的 Demo 页面 |

**「根据 PRD 变更」的变更来源识别：**

当用户说"根据我 PRD 的变更"时，变更可能来自多种来源，按以下顺序推断：
1. **当前对话中的修改**：本次会话中刚刚执行的 PRD 修改
2. **文件与会话内容对比**：重新读取 PRD 文件，与会话中已载入的 PRD 内容做对比，找出差异（用户经常在会话外手工修改 PRD 文件，这些改动不会自动反映到当前会话中）
3. **git diff**：运行 `git diff HEAD <PRD文件路径>` 查看未提交的变更记录

执行更新时：
1. 对比当前 PRD 与 Demo 内容，找出差异点
2. 按最小改动原则更新受影响的 Demo 文件
3. 完成后自动进入 Step 6，更新受影响页面的截图并回填 PRD

## Step 6: Demo 截图并回填 PRD

### 触发时机

以下情况自动执行本步骤：
- **首次生成 Demo 完成后**（Step 4 验证通过后）
- **用户明确要求更新 PRD 截图**（如「更新 PRD 里的截图」「帮我把截图塞进 PRD」）
- **Demo 更新后**（Step 5.2 中立即更新 Demo 后，或用户执行「更新 demo」指令后）

用户也可指定部分截图：
- `只更新 [页面名] 的截图` → 仅对指定页面截图并回填 PRD 对应章节

---

### 6.1 提取截图目标清单

从 Step 1 生成的页面清单出发，确认每个页面对应 PRD 中的哪个章节（功能点）：

```markdown
| # | Demo 页面文件 | 对应 PRD 章节 | 截图目标说明 |
|---|------------|--------------|------------|
| 1 | in-transit-tasks.html | 4.2 功能：审批任务转交 | 主列表页截图 |
| 2 | in-transit-tasks.html（转交弹窗） | 4.2 功能：审批任务转交 | 转交弹窗截图 |
| 3 | approval-data-mgmt.html | 4.4 功能：审批实例终止 | 终止按钮+弹窗截图 |
```

---

### 6.2 截图操作

使用以下工具截图（按优先级降级）：

**工具优先级：**
1. **`mcp__chrome-devtools__`**（首选）：直接操控浏览器，无需启动服务
2. **`document-skills:webapp-testing`**（降级）：使用 Playwright 脚本，适合 MCP Chrome DevTools 不可用时

**标准截图流程（chrome-devtools）：**

```
1. new_page(url) → 打开 Demo 页面
2. wait_for(text) → 等待页面加载完成
3. take_screenshot(filePath) → 截图保存到 assets/ 目录
4. 如需截取弹窗状态：
   - click(uid) → 触发弹窗
   - wait_for(text) → 等待弹窗出现
   - take_screenshot(filePath) → 截取弹窗状态
```

**降级截图流程（Playwright）：**

当 `mcp__chrome-devtools__` 工具不可用时，调用 `document-skills:webapp-testing` skill，使用 Python Playwright 脚本执行截图。注意：回填 PRD 时，脚本中的字符串匹配需处理文档中的特殊字符（如 `\xa0` 非断行空格），建议用正则或宽泛匹配。

**截图文件命名约定：**

```
docs/{PRD所在目录}/assets/
├── screenshot-{page-name}.png          # 页面主视图
├── screenshot-{page-name}-{modal}.png  # 弹窗状态
└── ...
```

**截图尺寸建议：**
- 列表/详情页：1440×900（桌面端标准宽度）
- 弹窗单独截图：保持页面完整，弹窗居中可见

---

### 6.3 回填 PRD

将截图插入 PRD 对应功能点章节的「界面原型与交互」部分：

**回填位置规则：**
- 新增页面的截图 → 插入对应功能点「界面原型与交互」末尾
- 弹窗截图 → 紧跟弹窗的 ASCII 原型图或交互说明之后
- 如果该功能点已有截图（旧截图）→ 替换为新截图

**回填格式：**

```markdown
![页面名称截图](assets/screenshot-page-name.png)
```

**示例：**
```markdown
#### 界面原型与交互

**页面布局**：
（ASCII 原型图...）

![在途审批任务列表截图](assets/screenshot-in-transit-tasks.png)

**批量转交结果弹窗**：
（ASCII 弹窗图...）

![批量转交结果弹窗截图](assets/screenshot-in-transit-tasks-batch-result.png)
```

---

### 6.4 完成确认

截图回填完成后，输出汇总：

```markdown
## 截图回填完成

| 截图文件 | 回填位置 |
|---------|---------|
| screenshot-in-transit-tasks.png | 4.2 功能：审批任务转交 - 界面原型 |
| screenshot-approval-data-mgmt.png | 4.4 功能：审批实例终止 - 界面原型 |
```
