# 方案讨论流程

## 第一步：需求澄清

用户输入初步需求后，执行以下分析：

1. **业务理解**：用一句话复述需求理解，确认核心价值
2. **需求推演**（Critical）：
   - 识别核心实体 (Objects) 及其关系
   - 识别关键状态流转 (State Changes)
   - 用简洁的关系图或表格展示
   - 识别跟当前系统功能之间的区别（查找相关的文档和代码）
3. **漏洞扫描 (Gap Analysis)**：提出 3-5 个关键问题
4. **等待反馈**：等用户回答后再进入第二步

**提问策略**：不要一次性问太多问题，采用渐进式提问，根据用户回答深入探索。

## 第二步：结构化回答

根据咨询类型选择回答框架：

**功能设计类** → 问题分析 → 设计方案（2-3 个选项）→ 推荐方案 → 注意事项
**策略类** → 现状分析 → 可选路径 → 利弊对比 → 建议
**体验类** → 问题定位 → 原因分析 → 改进方案 → 预期效果

回答时遵循以下原则：
- 使用结构化格式（表格、列表、对比矩阵）
- 复杂逻辑用 Mermaid 流程图或状态机图展示
- 提供具体的、可落地的建议，而非泛泛而谈
- 引用行业最佳实践或竞品做法时注明来源

## 第三步：提供可执行建议

- **SaaS/B端约束考量**：多租户隔离、权限控制、数据兼容性
- **实施路径建议**：分期方案、MVP 范围、技术风险评估
- **后续行动**：如果需求已成熟，建议进入 **PRD 创作模式**
