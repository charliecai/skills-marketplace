# PRD 创作流程

## 第一步：需求探索

与用户交互，明确以下关键信息：

1. **产品定位**
   - 产品名称和简介
   - 目标用户是谁？解决什么问题？核心价值是什么？

2. **业务目标**
   - 商业目标（营收、增长、留存等）
   - 用户目标（提升效率、改善体验等）

3. **功能范围**
   - 核心功能有哪些？优先级如何？（MVP vs 完整版）
   - 技术约束或限制？

4. **业务建模**（SaaS/B端场景适用）
   - 识别核心**实体 (Objects)** 及其关系
   - 识别关键的**状态流转 (State Changes)**
   - 用一句话复述对需求的理解，确认核心价值

5. **漏洞扫描 (Gap Analysis)**（SaaS/B端场景适用）
   - *必须询问*：历史存量数据如何处理？（如果涉及数据结构变更）
   - *必须询问*：涉及哪些角色的权限变动？
   - *必须询问*：异常场景如何处理？
   - *建议询问*：是否影响现有接口或对外 API？
   - *建议询问*：多租户场景下是否有差异化需求？

**提问策略**：
- 采用渐进式提问，根据用户回答深入探索
- SaaS/B端场景下，默认先完成业务建模和漏洞扫描后再进入文档生成
- 如果用户已提供充分完整的需求描述，可以跳过澄清直接创作

## 第二步：功能点拆分与组织

> 这一步在开始写文档之前完成，决定了 PRD 的骨架。

### 2.1 识别功能点

将需求拆分为**独立的功能点**。每个功能点应该是：
- **可独立理解**的一个完整功能单元
- **可独立验收**的一组变更
- 有明确的**输入→处理→输出**

**拆分粒度指导**：
- 太粗：「实现权限管理」→ 应拆为配置、校验、展示等功能点
- 太细：「按钮添加 tooltip」→ 应合并到所属功能点中
- 合适：「成员权限组新增数据范围配置」「审批数据管理列表按数据范围过滤」

### 2.2 识别流程图层次

拆分完功能点后，判断流程图的归属：

| 问题 | 判断 | 放置位置 |
|------|------|---------|
| 这个流程涉及多个功能点的协作吗？ | 是 → 整体流程 | `3.2 整体流程` |
| 这个流程是某个功能点内部的判断/处理逻辑吗？ | 是 → 功能内流程 | 该功能点的 `功能内流程` 子章节 |
| 这是一个状态机（实体的生命周期）吗？ | 是 → 核心概念 | `3.1 核心概念` |

**典型的整体流程**：用户完成操作A → 系统触发功能B → 功能C消费结果
**典型的功能内流程**：某接口被调用 → 校验权限 → 判断条件 → 执行逻辑 → 返回结果

### 2.3 确定文档骨架

根据拆分结果，确定功能需求章节的骨架：

```
3. 功能需求
  3.1 核心概念（如有跨功能点的共性概念）
  3.2 整体流程（如有跨功能点的流程）
  3.3 功能点1：[名称]
  3.4 功能点2：[名称]
  ...
  3.N 权限配置推导（如涉及权限变更）
```

## 第三步：内容撰写

### 结构原则

**功能点自包含原则**
- 每个功能点是一个完整的、自包含的信息单元
- 读者看完一个功能点的章节，就能完整理解该功能的所有方面
- 不需要跳转到其他章节去补全信息
- 功能点内的子章节（流程、UI、业务逻辑、验收等）按需裁剪，不必每个都写

**首次定义 + 后续引用原则**
- 跨功能共享的规则在首次出现的功能点中完整定义
- 后续功能点用章节链接引用，不重复完整描述
- 核心概念（3.1）仅放术语定义和前置知识（状态机、实体关系等不读不懂后文的内容），不堆积业务规则
- 判断标准：如果一条规则在 2 个以上功能点中出现，写完后确保只有一个完整定义，其余引用

**流程图分层原则**
- 整体流程（3.2）：展示全貌，回答"各功能点如何协作"
- 功能内流程：紧跟功能描述，帮读者建立该功能的整体认知地图
- 不要在整体流程中展开某个功能点的内部逻辑
- 不要在功能内流程中重复整体流程的信息

### 撰写原则

> 详细的写作风格规范参见 `references/PRD-WRITING-STYLE.md`。以下为核心原则摘要。

**首句即结论**
- 每个章节/小节的第一句话必须是该节的核心要点
- 读者扫读所有首句即可获得全文概貌

**认知递进原则**
- 业务逻辑按主流程 → 分支流程 → 边界与异常的顺序组织
- 读者先理解正常路径的完整行为，再看条件分支和差异，最后覆盖边界和异常
- 规则按关注点分组（显示与权限、核心行为、配置项等），不用扁平编号表格

**清晰性原则**
- 使用简洁明了的语言，避免模糊表述
- 用具体场景和示例说明需求
- 关键概念要有明确定义

**完整性原则**
- 覆盖所有必要的业务场景（正常流程、分支流程和异常流程）
- 包含边界条件和特殊情况
- 涉及数据结构变更时，说明存量数据的处理策略

**界面原型原则**
- 涉及新增页面或修改现有页面时，需提供界面原型描述
- 页面修改场景：明确所属页面、修改区域、变更内容
- 新增页面场景：描述页面入口、整体布局、内容区域和关键元素
- 界面描述应足够具体，可直接作为 AI 生成可交互 demo 的 prompt
- 界面描述中应包含关键视觉要素：状态标签的颜色语义、按钮类型（主按钮/次按钮/文字按钮/危险按钮）、布局方式
- 如果项目有设计系统描述文件或组件库索引，在 PRD 中引用，便于后续生成 Demo 时保持视觉一致性
- 生成 demo 后截图会自动补充到文档对应章节中，截图放置于 `assets/` 目录；后续 demo 更新时截图也会同步更新

**可执行性原则**
- 需求描述要具体可执行，不能过于抽象
- 提供具体的验收标准
- 包含必要的交互细节和视觉参考

**文风原则**（SaaS/B端场景适用）
- 专业、务实，直接写业务指标，避免空泛修饰语（如"旨在提升用户体验"）
- 使用标准 SaaS 术语（租户、实例、功能点、行权限、列权限）
- 复杂逻辑用 Mermaid 流程图或表格展示，不用纯文字描述

**决策背景原则**（SaaS/B端场景适用）
- PRD 中应包含关键设计决策的背景：为什么这样设计、有哪些备选方案、最终选择的理由
- 方案讨论阶段的竞品分析结论、客户原声引用应写入 PRD 对应章节（竞品分析、需求背景）
- 用户做出的明确决策（如选择方案A/B）及其理由，应在 PRD 的「设计决策记录」中留存
- 决策背景帮助后续维护者理解设计意图，避免未来无意间推翻已有决策

## 第四步：质量检查

创作完成后，除了按照 `references/REVIEW-CHECKLIST.md` 进行常规自查，还需要额外检查结构质量和写作质量：

### 结构质量检查清单

- [ ] **无跨章节重复**：同一规则是否只有一个完整定义？后续功能点是否用链接引用而非重复描述？
- [ ] **功能点自包含**：随机挑一个功能点单独阅读，能否完整理解该功能？是否需要跳到其他章节？
- [ ] **流程图归位**：整体流程是否在 3.2？功能内流程是否紧跟功能描述？
- [ ] **子章节按需裁剪**：功能点内的子章节是否都有实质内容？没有信息的子章节是否已删除？
- [ ] **核心概念不膨胀**：3.1 是否只放了术语定义和前置知识？是否有业务规则混入（应在首次出现的功能点中定义）？
- [ ] **认知递进**：业务逻辑是否按主流程 → 分支 → 边界与异常的顺序组织？

### 写作质量检查清单

- [ ] **去重审计**：扫描出现 2 次以上的规则描述，确保只有一个完整定义，其余为链接引用
- [ ] **首句扫读测试**：只读每节第一句话，能否在 30 秒内理解全文概貌？
- [ ] **AI 味检查**：对照 `references/PRD-WRITING-STYLE.md` 中的反模式清单，检查是否有空泛修饰语、填充句、扁平编号表格等问题
- [ ] **验收标准增量性**：验收标准是否只写了不能从业务逻辑直接推导的验证点？是否有复述规则的条目？

### 常规质量检查

- 逻辑是否清晰连贯？
- 需求是否完整无遗漏？
- 是否有矛盾或冲突？
- 是否可以直接交付开发？
