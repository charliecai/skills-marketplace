# 产品经理最佳实践

本文档聚焦于 HR SaaS / B端产品场景下的关键方法论和实践要点。通用 PM 知识（KANO 模型、用户访谈、A/B 测试等）不在此重复。

---

## 一、产品思维方法论

### 用户视角
- 用户真正的痛点是什么？
- 这个功能能为用户带来什么价值？
- 用户会如何使用这个功能？
- 用户使用过程中可能遇到什么问题？

### 数据驱动
- 定义关键指标（KPI）和目标值
- 考虑如何收集和分析数据
- 基于数据迭代优化

### 商业思维
- 投入产出比如何？
- 对品牌和口碑的影响？

### 技术理解
- 技术实现难度和资源需求
- 是否有技术风险？是否可以分期实现？

---

## 二、HR SaaS 需求分析要点

### 多租户场景特有考量
- **数据隔离**：功能变更是否影响租户间数据隔离？
- **配置差异**：不同租户是否需要差异化配置？
- **灰度策略**：新功能能否按租户分批上线？

### HR 业务域特有关注点
- **合规性**：是否涉及劳动法、社保、个税等法规要求？法规变更的适配能力
- **时效性**：薪资计算、考勤统计等有严格时间窗口
- **批量操作**：HR 场景常涉及批量入职、批量调薪、批量社保操作
- **审批链路**：几乎所有 HR 业务变更都涉及审批流程
- **历史追溯**：人事变更记录必须可追溯，不可物理删除

### 权限模型关注点
- **菜单权限**：功能入口是否需要新增/调整菜单
- **功能点权限**：具体操作按钮的可见性/可操作性
- **数据行权限**：按组织架构、汇报关系等维度控制数据可见范围
- **数据列权限**：薪资、身份证号等敏感字段的可见性控制

## 三、B端优先级判断框架

### RICE 评分法（B端适配）

```
RICE 分数 = (影响客户数 × 影响程度 × 信心指数) / 开发人天
```

- **影响客户数**：受影响的租户/企业数量
- **影响程度**：3（阻断业务）、2（降低效率）、1（体验不佳）、0.5（锦上添花）
- **信心指数**：100%（有客户投诉数据）、80%（有调研支撑）、50%（经验判断）
- **开发人天**：含前后端 + 测试

### B端特有优先级信号
- 使用卡点（P0）> 数据准确性问题（P0-P1）> 效率提升（P1）> 体验优化（P2）
- 影响薪资/社保/个税等涉及金额计算的问题，自动提升一级
- 涉及法规合规的需求，优先级不低于 P1

## 四、SaaS PRD 写作要点

### 闭环功能描述模式
每个功能点按以下结构描述，确保开发可直接实现：

```
触发条件 → 系统校验 → 后端逻辑 → 前端交互 → 后置处理
```

### 异常场景必查清单
- 并发操作：两人同时修改同一记录
- 跨月/跨年边界：薪资计算、考勤统计的月份切换
- 大批量数据：导入 1000+ 条记录的超时和分批处理
- 权限变更：操作过程中权限被收回
- 数据状态流转冲突：审批中的数据被编辑

### 文风要求
- 直接写业务指标，避免空泛修饰语（如"旨在提升用户体验"）
- 使用标准 SaaS 术语（租户、实例、功能点、行权限、列权限）
- 复杂逻辑用 Mermaid 流程图或表格展示，不用纯文字描述

## 五、存量数据处理策略

涉及数据结构变更时，必须回答以下问题：

1. **新增字段**：存量记录的默认值是什么？是否需要数据迁移脚本？
2. **修改枚举**：旧枚举值如何映射到新值？
3. **状态机变更**：处于旧状态的记录如何过渡？
4. **迁移时机**：发布前迁移 vs 发布后异步迁移 vs 灰度期间？
5. **回滚方案**：数据变更是否可回滚？

## 六、跨团队协作要点

### 与开发协作
- 需求阶段及早沟通技术可行性，听取技术方案建议
- 开发阶段及时答疑，快速响应变更
- 避免"需求不清晰就开始开发"和"变更随意"

### 需求宣讲结构
1. 背景和问题 → 2. 目标和价值 → 3. 方案概述 → 4. 核心功能详解 → 5. 排期计划 → 6. 答疑

### 向上汇报原则
- 结论先行，原因在后
- 提供可选方案 + 推荐方案 + 理由
- 用数据说话，说明潜在风险
