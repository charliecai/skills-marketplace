# PRD 写作风格规范

本文档定义 PRD 的写作风格标准。在 PRD 创作（MODE-2 Step 3）和质量检查（MODE-2 Step 4）中引用。

---

## 10 条写作规则

### 1. 首句即结论

每个章节/小节的第一句话必须是该节的核心要点。读者扫读所有首句即可获得全文概貌。

**反例**：
> 在当前的业务场景下，我们观察到审批流程在某些节点存在停留时间过长的问题，这对业务效率产生了一定影响。因此，我们需要引入催办机制。

**正例**：
> 当前审批系统缺乏催办机制，审批卡在节点时发起人和管理员没有手段推动进展。

### 2. 具体胜于抽象

用精确的数值、名称、操作替代模糊的修饰语。

**反例**：适当的频率限制、相关人数指标、提升用户体验
**正例**：10 分钟冷却期、花名册在职员工总数、审批处理时效从 3 天降至 1 天

### 3. 砍掉填充语

直接写要点，不用铺垫性语句开头。

**需要砍掉的**：
- "为了确保…"、"为了提升…" → 直接写做什么
- "需要注意的是…"、"值得指出的是…" → 直接写事实
- "在此基础上…"、"与此同时…" → 直接接下一个要点
- "旨在…"、"致力于…" → 直接写目标

### 4. 首次完整定义 + 后续引用

同一规则只在首次出现的功能点中完整描述。后续功能点用章节链接引用。

**反例**：每个功能点都写一遍"同一审批 10 分钟内只能催办一次，冷却期间提示…"

**正例**：
- 3.3 功能点1 中完整定义冷却机制
- 3.4 功能点2 中写"冷却机制同 [3.3 催办行为](#催办行为)"

### 5. 规则按关注点分组

不用扁平编号表格（`| # | 规则 |`），而是按关注点用加粗标题 + 列表分组。

**反例**：
```
| 1 | 审批中状态时显示催办按钮 |
| 2 | 发起人可见催办入口 |
| 3 | 点击后弹出选人组件 |
| 4 | 10分钟冷却 |
| 5 | 审批人已离职不发催办 |
```

**正例**：
```
**显示与权限**
- 审批中状态时显示催办按钮，已结束时隐藏
- 发起人、历史审批人、有权限管理员可见催办入口

**催办行为**
- 点击催办弹出选人组件，默认选中当前待审批人
- 冷却机制：同一审批 10 分钟内只能催办一次...

**边界与异常**
- 审批人已离职：不发催办，提示"..."
```

### 6. 表格用于结构化数据，列表用于规则

按信息结构选择格式：

| 信息类型 | 推荐格式 |
|---------|---------|
| ≥3 维度的结构化数据（配置项、竞品对比、API 清单） | 表格 |
| 分组业务规则 | 加粗分组标题 + 列表 |
| 操作-结果对 | 行内描述或短列表 |
| 仅两列且一列是序号的表格 | 直接用列表 |

### 7. 验收标准写增量

只写不能从业务逻辑直接推导的验证点。

**需要写的**：分支流程场景、跨模块联动、状态恢复（如续费后菜单恢复）、国际化验证
**不需要写的**：规则已明确说"审批中显示催办"，验收标准不必再写"审批中时可见催办按钮"

### 8. 避免 AI 典型用词

以下用词和句式在 PRD 中应避免：

| 避免 | 替代方式 |
|------|---------|
| "提升用户体验" | 写具体改善：减少操作步骤、缩短等待时间 |
| "确保…的同时…" | 拆成两个独立陈述 |
| "旨在…" | 直接写目的 |
| "赋能"、"抓手"、"打通" | 用具体动词：支持、提供、连接 |
| "此外"、"值得一提的是" | 直接写下一个要点 |
| "提供了…的能力" | 直接写"支持…" |

### 9. 前置动作，后置原因

句子以动作或结果开头，原因和背景放后面（或省略）。

**反例**：为了在模块到期时给用户明确提示，我们在前端路由层实现拦截，当用户点击已到期模块的菜单时跳转到到期提示页。

**正例**：模块到期后，菜单点击跳转到期提示页（前端路由层拦截）。

### 10. 主流程 → 分支 → 异常

业务逻辑按认知递进顺序组织：

1. **主流程**：正常路径下的完整行为，读者先理解"通常会发生什么"
2. **分支流程**：条件分支、特殊配置、不同角色的差异处理
3. **边界与异常**：离职、超时、接口失败等异常场景

读者在理解了主干之后再看枝叶，认知负担最小。

## 质量检查时的反模式清单

写完 PRD 后，逐项检查是否存在以下问题：

- [ ] 同一规则在 2 个以上位置完整描述（应只有一个定义，其余引用）
- [ ] 某节的第一句话是背景铺垫而非要点
- [ ] 功能规则用扁平编号表格，混杂了不同关注点
- [ ] 异常处理作为独立章节，4 列表格中有冗余列（场景和触发条件说的是同一件事）
- [ ] 验收标准复述了业务逻辑中已明确的规则
- [ ] 使用了"为了确保"、"旨在"、"提升用户体验"等填充语
- [ ] 业务逻辑没有按主流程→分支→异常的认知递进顺序组织
