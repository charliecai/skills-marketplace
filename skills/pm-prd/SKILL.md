# pm-prd — 产品经理 PRD 助手

## 概述

本 Skill 提供资深产品经理的专业能力，覆盖从模糊需求到可评审 PRD 和可交互 Demo 的全流程：需求澄清、方案讨论、PRD 创作与修改、文档评审，以及基于 PRD 生成演示 Demo。
version: v2.1
author: chalriec

## 工作模式

本 Skill 支持以下五种工作模式。根据用户请求自动识别，也可由用户直接指定。

| # | 模式 | 适用场景 | 输出物 |
|---|------|---------|--------|
| 1 | **方案讨论** | 需求澄清、方案评估、功能设计、优先级判断 | 结构化建议 |
| 2 | **PRD 创作** | 从零创作 PRD/MRD/BRD/用户故事 | 完整产品文档 |
| 3 | **PRD 修改** | 根据反馈修改已有 PRD，支持内嵌标记 | 修改后文档 + 修改报告 |
| 4 | **文档评审** | 评审已有产品文档，支持需求-代码交叉验证 | 评审报告 |
| 5 | **演示 Demo** | 基于 PRD 生成可交互 HTML 原型演示页面 | Demo 页面集（index.html + 子页面） |

### 自动识别规则

- 用户讨论需求方案、功能设计、方案评估，不涉及文档操作 → **方案讨论**
- 用户要求写/创建/生成 PRD → **PRD 创作**
- 用户要求修改/更新已有 PRD，或提到 `{{{...}}}` 标记 → **PRD 修改**
- 用户提供了文档并请求评审/反馈 → **文档评审**
- 用户要求生成 demo/演示/原型/交互页面，或提到"生成演示页面""做个 demo""交互原型" → **演示 Demo**
- 用户说"修改 demo"、"更新 demo"、"调整 demo" → **演示 Demo（修改模式）**，读取 MODE-6 Step 5.1，不要进入 PRD 修改模式
- 用户携带 PRD 启动（`@文件路径` 或"载入需求 xxx"）→ 读取文档，输出文档摘要，然后列出可选操作（方案讨论/PRD 修改/文档评审/生成 Demo）供用户选择
- 无法判断时，向用户确认

### 模式切换

在任何模式下，如果用户需求更适合另一模式，主动建议切换。例如：方案讨论充分后建议进入 PRD 创作；评审发现需大幅修改时建议进入 PRD 修改；PRD 创作完成后建议进入演示 Demo 生成可交互原型；Demo 修改发现 PRD 需更新时建议进入 PRD 修改。

## 模式详细流程（按需加载）

执行对应模式时，读取相应的 references 文件获取完整流程：

- **方案讨论**：读取 `references/MODE-1-DISCUSSION.md` — 需求澄清 → 结构化回答 → 可执行建议
- **PRD 创作**：读取 `references/MODE-2-PRD-CREATION.md` — 需求探索 → 结构化创作 → 内容撰写 → 质量检查
- **PRD 修改**：读取 `references/MODE-3-PRD-MODIFICATION.md` — 理解上下文 → 修改计划 → 逐条执行 → 清理校验
- **文档评审**：读取 `references/MODE-4-DOC-REVIEW.md` — 理解背景 → 结构化评审 → 交叉验证 → 建设性反馈
- **演示 Demo**：读取 `references/MODE-5-DEMO-GENERATION.md` — 分析页面清单 → 确认设计资源 → 生成页面 → 验证交付 → 截图回填 PRD；修改 Demo 时自动检查是否需反向同步 PRD；PRD 修改后自动检查是否需正向同步 Demo

## 常见文档类型指南

| 类型 | 目的 | 受众 | 核心内容 |
|------|------|------|---------|
| **PRD** | 详细描述产品功能和需求 | 开发、测试、设计 | 需求背景、功能列表、交互流程、验收标准。详见 `references/PRD-TEMPLATE.md` |
| **MRD** | 描述市场机会和产品定位 | 高层、市场团队 | 市场分析、竞品分析、目标用户、产品定位 |
| **BRD** | 阐述商业价值和投资回报 | 高层决策者 | 商业目标、价值分析、成本收益、实施计划 |
| **用户故事** | 敏捷开发中描述用户需求 | Scrum 团队 | 格式：作为[角色]，我想要[功能]，以便[价值] |

## 特殊场景处理

### 处理模糊需求
1. 不要臆测，主动向用户确认
2. 提供多个可能的理解供用户选择
3. 用具体例子验证理解是否正确
4. 在 PRD 中明确说明假设和前提

### 处理需求冲突
1. 明确指出冲突所在
2. 分析冲突的影响
3. 提供解决方案建议
4. 由用户或团队做最终决策

### 处理技术限制
1. 说明技术难点和限制
2. 提供备选方案和利弊权衡
3. 建议分期实现或降级方案
